What is CAP theorem? (Consistency-Availability-Partition Tolerance):

Consistency: This states that the data has to remain consistent after the execution of an operation in the database. 
		For example, post database updation, all queries should retrieve the same result.
Availability: The databases cannot have downtime and should be available and responsive always.
Partition Tolerance: The database system should be functioning despite the communication becoming 
					unstable.

Horizontal scaling vs Vertical scaling?

1. Horizontal scaling (Add more macnines): refers to the addition of more computing machines to the network that shares 
		the processing and memory workload across a distributed network of devices. In simple 
		words, more instances of servers are added to the existing pool and the traffic load is 
		distributed across these devices in an efficient manner.

2. Vertical scaling (Increase existing machine capacity): refers to the concept of upgrading the resource capacity such as increasing 
		RAM, adding efficient processors etc of a single machine or switching to a new machine 
		with more capacity. The capability of the server can be enhanced without the need for 
		code manipulation.

| **Category**              | **Horizontal Scaling**                                                                                        | **Vertical Scaling**                                                                   |
| ------------------------- | ------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| **Load Balancing**        | Needs a load balancer to distribute requests across multiple servers.                                         | No load balancer needed since all requests go to a single machine.                     |
| **Failure Resilience**    | High resilience: if one server fails, traffic can be routed to other servers.                                 | Low resilience: single point of failure; if the machine fails, the whole app is down.  |
| **Machine Communication** | Relies on **network communication** between servers, which adds latency.                                      | Uses **inter-process communication (IPC)** within the same machine → faster.           |
| **Data Consistency**      | May face data inconsistency issues (replication lag, sync problems across servers).                           | Consistency is simpler as all data is on one machine.                                  |
| **Limitations**           | More costly (multiple servers, more space, maintenance). But offers near-unlimited scaling by adding servers. | Limited by max hardware capacity (CPU, RAM, Disk). Beyond this, scaling is impossible. |
| **Cost Efficiency**       | Cost increases with more servers but can use commodity hardware.                                              | Cost-efficient initially, but upgrading to high-end hardware is expensive.             |
| **Use Cases**             | Large-scale, distributed apps (e.g., Google, Amazon, Facebook).                                               | Small to medium apps or monolithic architectures.                                      |

What do you understand by load balancing? Why is it important in system design?
Load balancing refers to the concept of distributing incoming traffic efficiently across a group 
of various backend servers. 

When a server goes down, the load balancer redirects traffic to the remaining available servers. 
When a new server gets added to the configuration, the requests are automatically redirected to it.

What is Sharding?
Sharding is a process of splitting the large logical dataset into multiple databases.
we have around 1TB of data present in the database, when we perform sharding, we divide the large 
1TB data into smaller chunks of 256GB into partitions called shards.


Partitioning = split a table inside one database instance into pieces (partitions) to speed queries/maintenance.

Sharding = split data across multiple database instances/servers (shards) to scale beyond a single box.

choose this if…

Monolith when:
	Team is small (≤10 engineers) or you want speed to first value.
	Domain is still evolving; requirements churn, frequent schema changes.
	You lack mature DevOps (CI/CD, observability, infra-as-code).
	You want simple transactions & queries across features in one DB.

Microservices when:
	Clear, stable bounded contexts (e.g., Billing, Catalog, Auth) with minimal cross-talk.
	You need independent scaling & deploys (e.g., high-write service vs low-latency read API).
	Org is multi-team with strong ownership per domain.
	You have operational maturity: CI/CD, tracing, metrics, service mesh, incident response.

Service Communication: REST, gRPC, message queues (Kafka, RabbitMQ).

When to use Rest vs gRPC:
Use REST for public/external APIs, browser/mobile clients, human-readability, CDN caching, and broad tooling.

Use gRPC for service-to-service calls where you need low latency, streaming, and strong contracts.

***********
Sql or No Sql:
SQL: structured schema, strong ACID transactions, great for relationships.
	 In SQL, scaling is often vertical (bigger CPU, RAM, disk on one server).
NoSQL: flexible schema, scales horizontally, optimized for specific patterns.
	Horizontal scaling means adding more machines (nodes/servers) to handle data and load.
	NoSQL databases are designed to distribute (shard) data across many nodes.
Example: MongoDB, Cassandra, DynamoDB automatically spread documents/rows by shard key.
	Queries route to the right node using the key → very fast lookups.
	This gives high availability (replicas) and fault tolerance (failover).
	Downside: no complex cross-node joins or transactions (need app-level handling).

is mysql rds support horizontal scaling?
Yes—for reads. RDS for MySQL supports read replicas to scale out read traffic (up to 15 replicas 
per source instance).

No—for writes. RDS MySQL remains single-writer; it doesn’t horizontally scale write throughput 
without sharding/partitioning at the application layer (or moving to engines designed for 
distributed writes).

Reliability & Fault Tolerance: Ensuring system functionality in the presence of failures.

To achive Reliability & Fault Tolerance do following:
Redundancy: Replication across zones/regions to remove single points of failure.
Failover Mechanisms: Active-passive and active-active setups.
			Active-Passive: one primary serves; a standby is idle/warm, takes over on failure.			
Monitoring & Alerts: Prometheus, Grafana, ELK stack, Sentry.
Data Durability: Write-ahead logs, backups, multi-region storage.

Case studies:
URL Shortner (bit.ly, TinyURL, ZipZy.in, etc) - System Design:
transforming lengthy web addresses into shorter, shareable links.

Steps:
1. Requirement Gathering:
	a. Functional requirements:
		Given a long URL, the service should generate a shorter and unique alias for it.
		When the user hits a short link, the service should redirect to the original link.
		Links will expire after a standard default time span.
	b. Non-Functional requirements:
		The system should be highly available. This is really important to consider because if the 
		service goes down, all the URL redirection will start failing.
		URL redirection should happen in real-time with minimal latency.
		Shortened links should not be predictable.

2. Capacity Estimation:
	Let's assume our service has 30M new URL shortenings per month. Let’s assume we store every URL shortening request (and associated shortened link) for 5 years. For this period the service will generate about 1.8 B records.
	30 million * 5 years * 12 months = 1.8B 

	Note: Let's consider we are using 7 characters to generate a short URL. These characters are 
	a combination of 62 characters [A-Z, a-z, 0-9] something like https://zpzy.in/redirecting?code=abXdef2.
	
URL Encoding Techniques to create Shortened URL:
To convert a long URL into a unique short URL we can use some hashing techniques like Base62 or 
MD5. We will discuss both approaches.

1. Base62 Encoding:
	Base62 encoder allows us to use the combination of characters and numbers which contains A-Z, a-z, 0–9 total( 26 + 26 + 10 = 62).
	So for 7 characters short URL, we can serve 62^7 ~= 3500 billion URLs which is quite enough in comparison to base10 (base10 only contains numbers 0-9 so you will get only 10M combinations).
	We can generate a random number for the given long URL and convert it to base62 and use the hash as a short URL id.
