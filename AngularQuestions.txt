1. What is a Decorator in Angular?
A decorator is a special kind of function that adds metadata to classes, properties, methods, or parameters. 
Angular uses decorators to define how things like components, services, and pipes should behave.
Decorators are prefixed with @.
Examples: @Component, @Injectable, @NgModule, @Input, @Output, etc.

| Feature               | `@Component` (Component Decorator)  | Decorator (General Concept)                 |
| --------------------- | ----------------------------------- | ------------------------------------------- |
| **What it is**        | A specific **type of decorator**    | A function that adds metadata               |
| **Purpose**           | Defines a **UI component**          | Used to **annotate classes/members**        |
| **Where it's used**   | On a class that defines a component | On classes, properties, parameters          |
| **Example**           | `@Component({...})`                 | `@Input()`, `@Injectable()`, etc.           |
| **Angular-specific?** | Yes                                 | No (Decorators are a TypeScript/ES feature) |
| **Part of UI?**       | Yes                                 | Not necessarily                             |

| Category  | Decorators                                                               |
| --------- | ------------------------------------------------------------------------ |
| Class     | `@Component`, `@Directive`, `@Pipe`, `@Injectable`, `@NgModule`          |
| Property  | `@Input`, `@Output`, `@HostBinding`, `@HostListener`, `@ViewChild`, etc. |
| Parameter | `@Inject`, `@Optional`, `@Self`, `@SkipSelf`, `@Host`                    |
| Custom    | User-defined (e.g., `@Log`, `@Cache`, etc.)                              |

What is a Guard in Angular?
An Angular Guard is used to control access to routes in your application. It decides whether a user 
can navigate to or away from a route, based on your logic (e.g., authentication, permissions, 
unsaved changes).
| Guard Type         | Interface          | Purpose                               |
| ------------------ | ------------------ | ------------------------------------- |
| `CanActivate`      | `CanActivate`      | Check if a route can be accessed      |
| `CanActivateChild` | `CanActivateChild` | Check if child routes can be accessed |
| `CanDeactivate`    | `CanDeactivate<T>` | Check if navigation away is allowed   |
| `Resolve`          | `Resolve<T>`       | Pre-fetch data before route loads     |
| `CanLoad`          | `CanLoad`          | Prevents lazy-loaded module loading   |

ng generate guard loginCheckGuard

export const loginCheckGuard: CanActivateFn = (route, state) => {
  const token = sessionStorage.getItem('authToken');
  if (!token) {
    // If token exists, redirect to home
    const router = inject(Router);  // 👈 use inject to get Router
    router.navigate(['/login']);
    return false;
  }
  // If no token, allow to access /login
  return true;
};

3. What is data binding?
Answer: Data binding in Angular is a technique to bind data between the component class and the template. 
Types include:
- Interpolation: {{ data }}
- Property binding: [property]="data"
- Event binding: (click)="handleClick()"
- Two-way binding: [(ngModel)]="data"

What is Property Binding in Angular?
Property binding in Angular is a one-way data binding technique used to bind the value of a 
component class property to an HTML element property.
<img [src]="imageUrl" />


4. What is a directive?
Answer: Directives are instructions in the DOM. There are three types:
- Component directives: Custom components.
- Structural directives: Change the DOM layout (e.g., *ngIf, *ngFor).
- Attribute directives: Change the appearance or behavior (e.g., ngClass, ngStyle).
<ul>
  <li *ngFor="let name of names">{{ name }}</li>
</ul>
<ul>
	<li *ngfor="let name of names; let i = index">{{i}}{{name}}</li>
<ul>
<p [ngClass]="currentClass">This text will be highlighted.</p>

Built-in Structural Directives:
| Directive           | Description                                                 |
| ------------------- | ----------------------------------------------------------- |
| `*ngIf`             | Adds/removes an element based on a condition                |
| `*ngFor`            | Loops over a list to render multiple elements               |
| `*ngSwitch`         | Adds/removes elements based on multiple possible conditions |
| `*ngSwitchCase`     | Defines one case in `ngSwitch`                              |
| `*ngSwitchDefault`  | Defines the fallback case in `ngSwitch`                     |
| `*ngTemplateOutlet` | Renders an `<ng-template>` dynamically                      |

//highlight.directive.ts
@Directive({
  selector: '[appHighlight]'
})
export class HighlightDirective {

  @Input('appHighlight') highlightColor: string = 'yellow';

  constructor(private el: ElementRef) {}

  @HostListener('mouseenter') onMouseEnter() {
    this.el.nativeElement.style.backgroundColor = this.highlightColor;
  }

  @HostListener('mouseleave') onMouseLeave() {
    this.el.nativeElement.style.backgroundColor = '';
  }
  @HostListener('click') onClick() {
    this.el.nativeElement.style.backgroundColor = "red";
  }
}

6. What are Angular lifecycle hooks?
Angular lifecycle hooks are special methods that Angular calls at specific moments in a component or 
directive's life — from creation to destruction.
Answer: Lifecycle hooks allow you to tap into key moments in a component's lifecycle:
- ngOnInit() – Called after the constructor and initialization.
- ngOnChanges() – Called when input properties change.
- ngOnDestroy() – Called just before the component is destroyed.

Here’s the typical sequence for a component:
constructor → ngOnChanges → ngOnInit → ngDoCheck → ngAfterContentInit → 
ngAfterContentChecked → ngAfterViewInit → ngAfterViewChecked → ngOnDestroy

| Hook                      | When it runs                                           | Common Use Case                         |
| ------------------------- | ------------------------------------------------------ | --------------------------------------- |
| `constructor()`           | When the class is instantiated                         | Basic setup (no DOM access)             |
| `ngOnChanges()`           | When input properties change (`@Input()`)              | Respond to input changes                |
| `ngOnInit()`              | Once after the first `ngOnChanges()`                   | Fetch data, init logic                  |
| `ngDoCheck()`             | Runs during every change detection cycle               | Custom change detection                 |
| `ngAfterContentInit()`    | After projecting external content (`ng-content`)       | Logic on slotted content                |
| `ngAfterContentChecked()` | After every check of projected content                 | Watch projected content updates         |
| `ngAfterViewInit()`       | After component's view and child views are initialized | Access template elements (`@ViewChild`) |
| `ngAfterViewChecked()`    | After every check of component’s view                  | DOM-dependent updates                   |
| `ngOnDestroy()`           | Just before the component is removed                   | Cleanup: unsubscribe, clear timers      |

ngAfterContentInit is a lifecycle hook that is called after Angular projects external content 
(passed using <ng-content>) into your component.

8. What is a module in Angular?

What is ahead-of-time (AOT) and just-in-time (JIT)
Angular supports two ways of compiling your templates and components:
| Mode    | Full Name     | When It Happens                 | Use Case    |
| ------- | ------------- | ------------------------------- | ----------- |
| **AOT** | Ahead-of-Time | Before app runs (during build)  | Production  |
| **JIT** | Just-in-Time  | In the browser (when app loads) | Development |

Ahead-of-Time (AOT) Compilation:
✔️ Templates and TypeScript are compiled into JavaScript before the browser runs the app.
Key Features:
Happens at build time
Generates optimized JS
Smaller, faster app
Detects template errors early
Used by default in Angular production build

ng build --configuration=production
Uses: @angular/compiler-cli instead of @angular/compiler in the browser

How to Build and Run an Angular App in Production Mode:
ng build --configuration=production
This generates an optimized, minified version of your app inside the dist/ folder:
Angular's build output is just static HTML/CSS/JS — so you need a web server to host it.
Here are common ways to serve it:
npm install -g http-server
http-server dist/your-app-name


Just-in-Time (JIT) Compilation:
Angular compiles HTML templates in the browser at runtime when the app loads.
Key Features:
Happens in browser
Slower startup
Bigger JS bundle
Easy for development (hot reload, fast rebuilds)
Detects template errors at runtime

ng serve   # default uses JIT

11. What is lazy loading in Angular?
Answer: Lazy loading loads feature modules only when they're needed, reducing the initial load time 
of the application. It’s implemented using the loadChildren property in routes.

ng generate module admin --route admin --module app.module
const routes: Routes = [
  {
    path: 'admin',
    loadChildren: () =>
      import('./admin/admin.module').then(m => m.AdminModule)
  }
];

Difference Between Component and Module:
| Aspect            | **Component**                                           | **Module**                                                       |
| ----------------- | ------------------------------------------------------- | ---------------------------------------------------------------- |
| **Definition**    | A component controls a part of the UI (view).           | A module groups related components, directives, services, etc.   |
| **Purpose**       | Used to build and control a specific UI section.        | Organizes the application into cohesive blocks of functionality. |
| **Decorator**     | `@Component()`                                          | `@NgModule()`                                                    |
| **Contains**      | Template (HTML), Class (logic), CSS                     | Components, directives, pipes, and services                      |
| **Example Usage** | `<app-header></app-header>` renders a header component. | `AppModule`, `AdminModule`, `UserModule`, etc.                   |
| **Bootstrap**     | Can’t bootstrap an app directly.                        | Root module (`AppModule`) bootstraps the app.                    |
| **Reusability**   | Represents a reusable UI element.                       | Can be lazy-loaded and reused as a feature module.               |

12. What is a reactive form vs. a template-driven form?
Answer:
- Reactive Form: More control, scalable, uses FormGroup, FormControl.
- Template-driven Form: Simpler, uses ngModel, suitable for simple forms.

| Feature        | Reactive Form              | Template-Driven Form          |
| -------------- | -------------------------- | ----------------------------- |
| Code Location  | Component class            | HTML Template                 |
| Syntax Style   | Imperative (code-driven)   | Declarative (template-driven) |
| Form Creation  | `FormGroup`, `FormControl` | `ngModel`                     |
| Suitable For   | Complex forms              | Simple forms                  |
| Unit Testing   | Easier                     | Harder                        |
| Learning Curve | Steeper                    | Easier                        |

<!-- Reactive Form -->
export class AppComponent {
  myForm = new FormGroup({
    name: new FormControl('', Validators.required),
    email: new FormControl('')
  });
  
<form [formGroup]="myForm" (ngSubmit)="onSubmit()">
  <input formControlName="name">
  <input formControlName="email">
  <button type="submit">Submit</button>
</form>


<!-- Template-Driven Form -->
<form #formRef="ngForm" (ngSubmit)="onSubmit(formRef)">
  <input name="name" [(ngModel)]="name" required>
  <input name="email" [(ngModel)]="email">
  <button type="submit">Submit</button>
</form>

How to Pass Data from Parent to Child in Angular:
use Input properties with the @Input() decorator in the child.

// child.component.ts
@Component({
  selector: 'app-child',
  template: `<p>Message from parent: {{ message }}</p>`
})
export class ChildComponent {
  @Input() message: string = '';
}

// parent.component.ts
@Component({
  selector: 'app-parent',
  template: `<app-child [message]="parentMessage"></app-child>`
})
export class ParentComponent {
  parentMessage = 'Hello from Parent!';
}

Child to parent:
<!-- child.component.html -->
<input [(ngModel)]="inputValue" (input)="sendValue()">

// child.component.ts
@Output() valueChange = new EventEmitter<string>();
inputValue: string = '';

sendValue() {
  this.valueChange.emit(this.inputValue);
}

<!-- parent.component.html -->
<app-child (valueChange)="onValueChanged($event)"></app-child>
<p>Value from child: {{ childValue }}</p>

export class ParentComponent {
  childValue = '';

  onValueChanged(newVal: string) {
    this.childValue = newVal;
  }
}

What is <app-root></app-root>:
<app-root> is the custom HTML tag (a.k.a. component selector) that tells Angular where to load and render your main root component.

What is <router-outlet></router-outlet> in Angular?
<router-outlet> is a built-in Angular directive (not a regular HTML tag).
It acts as a placeholder that Angular dynamically replaces with the routed component based on the current URL.

Difference between component and directive
A directive adds behavior to existing DOM elements. There are two types:
1) Attribute Directive -> ngClass, ngStyle
b) Structural Directive -> *ngIf, *ngFor

What is NgRx:
NgRx (short for Angular Reactive Extensions) is a state management library for Angular apps.
It helps you manage complex application state using a pattern similar to Redux (from React world), powered by RxJS (Reactive programming).

Why Use NgRx?
Angular apps grow → state becomes hard to track.
Data flows between many components and services.
NgRx centralizes state, making it:
Predictable
Traceable (with DevTools)
Testable
Scalable

| Concept      | Description                                                      |
| ------------ | ---------------------------------------------------------------- |
| **Store**    | Global container holding all state                               |
| **Action**   | Events that describe what happened (`LOGIN`, `LOAD_USERS`, etc.) |
| **Reducer**  | Pure function to update state based on action                    |
| **Effect**   | Handles side-effects (like API calls) using RxJS                 |
| **Selector** | Efficient way to read state from the store                       |
| **Dispatch** | Sends an action to the store                                     |


@ViewChild
In Angular, @ViewChild() is a decorator that allows you to access a child component, directive, or DOM element 
from your component class.

It gives you a reference to:
A component (child)
A directive
A template variable (#templateRef)
A DOM element


@ViewChild(ChildComponent) childComp!: ChildComponent;

Or with a template reference variable:
<input #myInput>
@ViewChild('myInput') inputRef!: ElementRef;

Accessing a DOM element:
<input #inputBox type="text">
<button (click)="logInput()">Log Input</button>

@ViewChild('inputBox') inputBoxRef!: ElementRef;

logInput() {
  console.log(this.inputBoxRef.nativeElement.value);
}

Accessing a Child Component:
<app-child #child></app-child>
<button (click)="callChildMethod()">Call</button>

@ViewChild('child') childComp!: ChildComponent;

callChildMethod() {
  this.childComp.someMethod();
}

The @ViewChild reference becomes available after the view is initialized, so use it inside ngAfterViewInit().


How to communicate between 2 unrelated components:
using a shared service with RxJS Subjects or Observables.
@Injectable({ providedIn: 'root' })
export class SharedService {
  private messageSource = new BehaviorSubject<string>('Initial Message');
  currentMessage$ = this.messageSource.asObservable();

  changeMessage(message: string) {
    this.messageSource.next(message);
  }
}
 
// component-a.component.ts

@Component({
  selector: 'app-component-a',
  template: `<button (click)="sendMessage()">Send Message</button>`
})
export class ComponentA {
  constructor(private sharedService: SharedService) {}

  sendMessage() {
    this.sharedService.changeMessage('Hello from Component A!');
  }
}

@Component({
  selector: 'app-component-b',
  template: `<p>Received: {{ message }}</p>`
})
export class ComponentB implements OnInit {
  message: string;

  constructor(private sharedService: SharedService) {}

  ngOnInit() {
    this.sharedService.currentMessage$.subscribe(msg => this.message = msg);
  }
}
