JVM consists of a few memory storages as mentioned below:

1. Class(Method) Area: stores class-level data of every class such as the runtime constant pool, field, and method data, and the code for methods.
2. Heap: Objects are created or objects are stored. It is used to allocate memory to objects during run time.
3. Stack: stores data and partial results which will be needed while returning value for method and performing dynamic linking
4. Program Counter Register: stores the address of the Java virtual machine instruction currently being executed.
5. Native Method Stack: stores all the native methods used in the application.

clone():
clone() is a method in the Object class.
It creates a copy of the current object and returns it.
It performs a shallow copy by default (not deep copy).
*** It's a protected method, so it must be overridden and made public in your class.

class Person implements Cloneable {
    String name;
    int age;

    Person(String name, int age) {
        this.name = name;
        this.age = age;
    }   
    
    @Override
    public Object clone() throws CloneNotSupportedException {
        return super.clone();
    }
}

Person p1 = new Person("John", 25);
Person p2 = (Person) p1.clone();
System.out.println(p1.name);  // John
System.out.println(p2.name);  // John
System.out.println(p1 == p2); // false (different objects)
		
Cloneable interface:
A marker interface (it has no methods).
If a class does not implement Cloneable, calling clone() on its object throws CloneNotSupportedException.
It tells the JVM:
“This class allows cloning.”

Shallow Copy:
Copies the object and its references.
Nested objects are not cloned — both original and copy share the same inner objects.
Made using:
✅ Object.clone() (default),
✅ Copy constructors (sometimes),
❌ Not safe if inner objects are mutable.


Deep copy:
Creates a completely independent copy, including all nested objects.
Each object in the structure is cloned recursively.
You must manually implement this or use tools like serialization or libraries like Apache Commons Lang.

class Address {
    String city;
    Address(String city) {
        this.city = city;
    }
}

class Person implements Cloneable {
    String name;
    Address address;

    Person(String name, Address address) {
        this.name = name;
        this.address = address;
    }

    // 🔹 Shallow copy
    public Object clone() throws CloneNotSupportedException {
        return super.clone();
    }

    // 🔸 Deep copy
    public Person deepClone() {
        return new Person(this.name, new Address(this.address.city));
    }
}



How to make immutable object:
Make the class final – so it can't be extended.
Make all fields private and final – to prevent modification.
No setters – only getters.
Initialize all fields via constructor.
For mutable fields, return a defensive copy instead of original reference.


public final class Employee {
    private final String name;
    private final int id;
    private final Date joinDate;  // Mutable field

    public Employee(String name, int id, Date joinDate) {
        this.name = name;
        this.id = id;
        this.joinDate = new Date(joinDate.getTime()); // Defensive copy
    }

    public String getName() {
        return name;
    }

    public int getId() {
        return id;
    }

    public Date getJoinDate() {
        return new Date(joinDate.getTime()); // Defensive copy
    }
}

How can you synchronize an ArrayList in Java?

3 Common Ways to Synchronize an ArrayList

1. 
	List<String> list = Collections.synchronizedList(new ArrayList<>());
	// Thread-safe access example
	synchronized (list) {
		list.add("Apple");
	}

	You must use synchronized(list) when iterating to avoid ConcurrentModificationException.

2. Using CopyOnWriteArrayList -> without needing manual synchronization.
	List<String> list = new CopyOnWriteArrayList<>();
	list.add("Banana");
	
	✅ Safe to read and write concurrently
	❌ Not efficient for frequent writes (creates a new copy every time)

3. Manual Synchronization (not recommended unless needed)
	List<String> list = new ArrayList<>();

	synchronized (list) {
		list.add("Mango");
	}

	You have to synchronize manually everywhere, which is error-prone.

Why do we need a synchronized ArrayList when we have Vectors (which are synchronized) in Java?
ArrayList is faster than Vectors.
ArrayList supports multithreading whereas Vectors only supports single-thread use.
ArrayList is safer to use, as Vectors supports single threads and individual operations are less safe and take longer to synchronize.
Vectors are considered outdated in Java because of their synchronized nature.

how elements are stored in memory for both regular arrays and ArrayLists in Java? 

array in Java are stored in contiguous memory locations, meaning that each element is stored in a 
sequential block of memory

ArrayList elements are also stored in contiguous memory locations, similar to arrays. 
However, when an ArrayList reaches its capacity and more elements need to be added, a new, 
larger underlying array is created. The elements from the old array are then copied to the new one. 
This process ensures that the ArrayList can grow dynamically while keeping the elements in 
contiguous memory locations.

FailFast iterator and FailSafe iterator:
A FailFast iterator is an iterator that throws a ConcurrentModificationException if it detects that the 
underlying collection has been modified while the iterator is being used.
iterator for a HashMap is FailFast.

HashMap<Integer, String> map = new HashMap<>();
map.put(1, "one");
map.put(2, "two");

Iterator<Map.Entry<Integer, String>> iterator = map.entrySet().iterator();
while (iterator.hasNext()) {
    Map.Entry<Integer, String> entry = iterator.next();
    // this will throw a ConcurrentModificationException
    if (entry.getKey() == 1) {
        map.remove(1); 
    }
}

A FailSafe iterator does not throw a ConcurrentModificationException if the underlying collection is 
modified while the iterator is being used.
ConcurrentHashMap is FailSafe.
ConcurrentHashMap<Integer, String> map = new ConcurrentHashMap<>();
map.put(1, "one");
map.put(2, "two");

Iterator<Map.Entry<Integer, String>> iterator = map.entrySet().iterator();
while (iterator.hasNext()) {
    Map.Entry<Integer, String> entry = iterator.next();
    // this will not throw an exception
    if (entry.getKey() == 1) {
        map.remove(1);
    }
}

Runtime exceptions:
Unlike checked exceptions, runtime exceptions do not require a declaration in the throws clause or 
capture in a try-catch block. However, handling runtime exceptions is advisable in order to provide 
meaningful error messages and prevent a system crash.

Checked Exception and Unchecked Exception:
Checked Exceptions are the exceptions that are checked during compile time of a program. In a program, 
if some code within a method throws a checked exception, then the method must either handle the exception 
or must specify the exception using the throws keyword. 

Checked exceptions are of two types: 
Fully checked exceptions: all its child classes are also checked, like IOException, and InterruptedException.
Partially checked exceptions: some of its child classes are unchecked, like an Exception. 

Unchecked Exception:
Unchecked are the exceptions that are not checked at compile time of a program. Exceptions under Error 
and RuntimeException classes are unchecked exceptions, everything else under throwable is checked. 


Exception rule for method override in java:
| Can it...?                                     | Allowed?                  |
| ---------------------------------------------- | ------------------------- |
| Declare **no exception**                       | ✅ Yes                     |
| Declare the **same exception**                 | ✅ Yes                     |
| Declare a **subclass** of that exception       | ✅ Yes                     |
| Declare a **broader or new checked exception** | ❌ No (Compile-time error) |

class Parent {
    void show() throws IOException { }
}

class Child extends Parent {
    @Override
    void show() throws FileNotFoundException { } // ✅ Allowed (subclass of IOException)
}

class AnotherChild extends Parent {
    @Override
    void show() throws Exception { } // ❌ Compile-time error (broader exception)
}


finally rules:

| Rule # | Description                                                                            |
| ------ | -------------------------------------------------------------------------------------- |
| 1️⃣    | `finally` **always executes**, even if an exception is thrown or caught.               |
| 2️⃣    | Executes even if there’s a `return`, `break`, or `continue` inside `try` or `catch`.   |
| 3️⃣    | If the JVM shuts down (`System.exit()`), `finally` does **not** execute.               |
| 4️⃣    | `finally` is optional but **recommended** when using `try`.                            |
| 5️⃣    | If both `try`/`catch` and `finally` have `return`, the `finally`'s `return` overrides. |
| 6️⃣    | `finally` is **not executed** if the thread is killed or JVM crashes.                  |


finalize:
finalize() is a method in the Object class that Java calls just before an object is garbage collected, 
giving it a chance to clean up resources (like closing a file or releasing memory).

@Override
protected void finalize() throws Throwable {
    // cleanup code
    System.out.println("Object is being garbage collected");
}

Why finalize() is Deprecated (Java 9+)?
Unpredictable: No guarantee when (or if) it will be called.
Performance issue: GC delays object removal if finalize() is present.
Safer alternatives: try-with-resources, AutoCloseable, or explicit close() methods.

life cycle of the thread:
New: The thread has been created but has not yet started.
Runnable: The thread is running, executing its task, or is ready to run if there are no other higher-priority threads.
Blocked: The thread is temporarily suspended, waiting for a resource or an event.
Waiting: The thread is waiting for another thread to perform a task or for a specified amount of time to elapse.
Terminated: The thread has completed its task or been terminated by another thread.

main thread:
main thread is considered the parent thread of all the other threads
The main thread is automatically created when the program starts running. 
This thread executes the main method of the program.

daemon thread:
daemon thread in Java is a low-priority thread that is used to perform background operations or 
tasks which are used to perform continuously. such as Garbage collection, Signal dispatches, 
Action listeners, etc. Daemon threads in Java have lower priority than user threads, 
which means they can only execute when no user threads are running.

Why Garbage Collection is necessary in Java?
to avoid memory leaks which can cause the program to crash and become unstable.
focus on the development of the application instead of managing memory resources and worrying about memory leakage.


What is the drawback of Garbage Collection?
1. it works to clear the memory which slows down the performance of the application. 
2. The Process of Garbage collection is non-deterministic which makes it difficult to predict when 
	garbage collection occurs which causes unpredictable behavior in applications. 
	For Example, if we write any program then it is hard for programmers to decide if the issue is caused 
	by garbage collection or by any other factors in the program. 
3. Garbage collection can also increase memory usage if the program creates and discards a lot of 
	short-lived objects.

difference between a minor, major, and full garbage collection:
1. Minor garbage collection: used to collect and reclaim memory that is used by short-lived objects
2. Major garbage collection: used to collect and reclaim memory that is used by long-lived objects 
      (objects that survive multiple minor garbage collections and are promoted to the old generation).

3. Full garbage collection: During full garbage collection, memories from all generations are collected and 
	reclaimed, including memories of young and old. 

What is a memory leak, and how does it affect garbage collection:
caused by not closing resources properly, holding onto object references longer than necessary, 
or creating too many objects unnecessarily. 
These useless objects which do not provide any value to the program are known as Memory leaks.
Memory leaks can impact garbage collection negatively by preventing the garbage collector from reclaiming 
unused memory. This behavior will lead to slow performance or sometimes system failure.

Atomic classes:
They are used when multiple threads need to update a shared variable (like an integer counter or object 
reference) without using synchronization (synchronized) or explicit locks, thereby improving performance 
and scalability.
| Class Name                | Description                                          |
| ------------------------- | ---------------------------------------------------- |
| `AtomicInteger`           | For atomic operations on `int` values                |
| `AtomicLong`              | For atomic operations on `long` values               |
| `AtomicBoolean`           | For atomic operations on `boolean` values            |
| `AtomicReference<V>`      | For atomic operations on object references           |
| `AtomicIntegerArray`      | For atomic operations on arrays of `int`             |
| `AtomicLongArray`         | For atomic operations on arrays of `long`            |
| `AtomicReferenceArray<V>` | For atomic operations on arrays of object references |


AtomicInteger counter = new AtomicInteger(0);
// Atomically increments and returns new value
counter.incrementAndGet();
// Atomically gets the value
int value = counter.get();
// Atomically sets a new value
counter.set(5);
// Atomically compares and sets the value
counter.compareAndSet(5, 10); // returns true if value was 5

These classes use low-level atomic CPU instructions like Compare-And-Swap (CAS) via the Unsafe 
class under the hood, ensuring updates happen atomically without locks.
