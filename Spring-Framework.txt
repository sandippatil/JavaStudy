Explain Dependency Injection(DI) and its types?

Dependency Injection (DI) is a design pattern used in Spring Framework (and other frameworks) 
to inject objects (dependencies) into a class instead of letting the class create them.

increasing developers' productivity by reducing boilerplate code.

Why Use DI?
Loose coupling between classes
Better testability
Easier maintenance and scalability

Types of Dependency Injection in Spring:
1. Constructor Injection (Recommended)
2. Setter Injection
3. Field Injection (Not recommended for testing)

Why Constructor Injection is Recommended
1. Immutability: You can declare dependencies as final, ensuring they can’t be changed once set.
	private final UserService userService;
	Helps prevent bugs
	Encourages clean, thread-safe design
2. Clear Dependencies:
	Dependencies are explicitly required in the constructor.
	No hidden dependencies = better maintainability.
	Easy to understand, document, and test
3. Enforced at Compile Time:
	Missing required dependencies cause errors immediately at startup.
	Unlike setter or field injection, which can silently fail or leave a bean partially initialized.
4. Testability:
	Works naturally with unit tests and mocks.
	No need for Spring container in most unit tests.

| Type             | Problem                                            |
| ---------------- | -------------------------------------------------- |
| Field Injection  | Hidden dependencies, hard to test, reflection used |
| Setter Injection | Allows partial construction, mutable state         |

Explain Spring Beans and their scopes:
Objects managed by the Spring container are called Beans..defined by configuration and annotated with @Component.

Scopes: Define the lifecycle and lifetime of a bean:
Singleton: Single instance throughout the application.
Prototype: A new instance is created for each request.
Request: A new instance is created for each HTTP request.

Session scope:
Spring stores the bean in the HTTP session (like a Map with bean names as keys).
The bean instance is reused for all requests in the same session.

What is circular dependency and how to resolve?
A circular dependency happens when two or more Spring beans depend on each other in a loop, 
creating a cycle that Spring can't resolve during injection.

1. Use Setter or Field Injection -> Spring creates beans first, then wires dependencies — no cycle
	@Component
	public class A {
		private B b;

		@Autowired
		public void setB(B b) {
			this.b = b;
		}
	}
	
2. Use @Lazy Annotation
@Component
public class A {
    private final B b;

    @Autowired
    public A(@Lazy B b) {
        this.b = b;
    }
}

3. Redesign Your Code (Best Practice):
	Try to:
		Extract shared logic into a third bean (C) that both A and B depend on		

Bean life cycle in a Spring Bean Factory Container:
1. Bean Instantiation -> Spring creates the bean using a constructor or factory method.
2. Populate Properties (Dependency Injection)
3. Spring injects dependencies (@Autowired, @Value, etc.).
4. Bean Name Awareness (optional)
	If the bean implements BeanNameAware, Spring calls setBeanName().
5. Bean Factory Awareness (optional)
	If the bean implements BeanFactoryAware, Spring calls setBeanFactory().
6. Application Context Awareness (optional)
	If the bean implements ApplicationContextAware, Spring calls setApplicationContext().
7. Pre-Initialization Hooks
	Custom pre-init logic from BeanPostProcessor.postProcessBeforeInitialization()

8. InitializingBean / @PostConstruct
	afterPropertiesSet() if implementing InitializingBean Or, method annotated with @PostConstruct

9. Post-Initialization Hooks
	BeanPostProcessor.postProcessAfterInitialization()

10. Bean Is Ready for Use

11. Destruction Phase
	@PreDestroy or DisposableBean.destroy()
	Occurs when the context is closed or bean is removed (except prototype beans)

Autowiring and its types:
Types of Autowiring:
No auto wiring: Setter or constructor-based dependency injection.
By name: Matches bean names with property names for injection.
By type: Matches bean types with property types for injection.
Constructor: Injects dependencies through the bean's constructor.

Annotation-Based Autowiring (Modern Approach)
Spring Boot and annotation config use:
@Autowired → By Type
@Qualifier("beanName") → By Name
@Inject → By Type (JSR-330)
@Resource(name="beanName") → By Name

How to Activate Spring Profiles:
1. Set Environment Variable
	export SPRING_PROFILES_ACTIVE=dev
2. Set in Application Server (e.g., Tomcat)
	-Dspring.profiles.active=dev
3. Set in application.properties (fallback)
	spring.profiles.active=${ENV_PROFILE:dev}
	This uses environment variable ENV_PROFILE
	Defaults to dev if not set
4. Use .env File (Spring Boot + Docker)
	SPRING_PROFILES_ACTIVE=prod
	Dockerfile should pass the env or use --env-file .env

5. Set in Kubernetes (Spring Boot + K8s)
	deployment.yaml
	env:
	  - name: SPRING_PROFILES_ACTIVE
		value: prod

What is Spring AOP and proxy pattern?
Aspect-oriented programming (AOP) is a design pattern that helps us manage aspects 
like logging, security, and transaction management in applications.

Spring AOP provides an implementation of AOP using dynamic proxies.
The proxy pattern is a software design pattern that creates a proxy object that intercepts incoming 
requests and controls access to another object before reaching the bean. 
In Spring AOP, dynamic proxies are used to implement aspects.


Explain key components of AOP:
| Component         | Description                                                             |
| ----------------- | ----------------------------------------------------------------------- |
| **Aspect**        | A class that contains cross-cutting logic (a concern like logging)      |
| **Join Point**    | A specific point in execution (like a method call) where advice can run |
| **Advice**        | The action taken at a join point (like "log before this method")        |
| **Pointcut**      | A rule that matches join points (e.g., all methods in a package)        |
| **Weaving**       | The process of applying aspects to target objects                       |
| **Target Object** | The object whose method execution is being advised                      |
| **Proxy**         | A dynamic wrapper that applies advice around the target object          |


@Aspect
@Component
public class LoggingAspect {

    @Before("execution(* com.example.demo.service.*.*(..))")
    public void logBeforeMethod(org.aspectj.lang.JoinPoint joinPoint) {
        System.out.println("🔍 [Before] Calling: " + joinPoint.getSignature().getName());
    }

    @After("execution(* com.example.demo.service.*.*(..))")
    public void logAfterMethod(org.aspectj.lang.JoinPoint joinPoint) {
        System.out.println("✅ [After] Completed: " + joinPoint.getSignature().getName());
    }

    @AfterReturning(pointcut = "execution(* com.example.demo.service.*.*(..))", returning = "result")
    public void logAfterReturn(org.aspectj.lang.JoinPoint joinPoint, Object result) {
        System.out.println("✅ [AfterReturning] Method returned successfully");
    }

    @AfterThrowing(pointcut = "execution(* com.example.demo.service.*.*(..))", throwing = "ex")
    public void logAfterThrow(org.aspectj.lang.JoinPoint joinPoint, Throwable ex) {
        System.out.println("❌ [AfterThrowing] Exception: " + ex.getMessage());
    }

    @Around("execution(* com.example.demo.service.*.*(..))")
    public Object logAround(ProceedingJoinPoint pjp) throws Throwable {
        String methodName = pjp.getSignature().getName();
        System.out.println("⏱️ [Around-Before] Starting method: " + methodName);
        long start = System.currentTimeMillis();

        try {
            Object result = pjp.proceed(); // proceed to actual method
            long end = System.currentTimeMillis();
            System.out.println("✅ [Around-After] Finished method: " + methodName + " in " + (end - start) + "ms");
            return result;
        } catch (Throwable ex) {
            System.out.println("❌ [Around] Exception in method: " + methodName);
            throw ex; // rethrow to let @AfterThrowing and caller handle it
        }
    }
}

Filters vs Interceptors:

| Feature               | **Filter** (Servlet)                         | **Interceptor** (Spring MVC)                    |
| --------------------- | -------------------------------------------- | ----------------------------------------------- |
| **Level**             | Servlet-level (javax.servlet)                | Spring MVC-level                                |
| **Interface**         | `javax.servlet.Filter`                       | `HandlerInterceptor`                            |
| **Applied To**        | All requests (static, REST, JSP, etc.)       | Only **Spring MVC** mapped requests             |
| **Use Case**          | Logging, security headers, encoding          | Authentication, logging, modifying model/view   |
| **Access to Handler** | ❌ No                                         | ✅ Yes (has access to controller info)           |
| **Ordering**          | Set via `@Order` or `FilterRegistrationBean` | Registered in `WebMvcConfigurer`                |
| **Scope**             | Broader (works even outside Spring context)  | Narrow (only Spring DispatcherServlet requests) |


@Component
public class RequestLoggingFilter implements Filter {

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        System.out.println("🌐 Filter - Request received");
        chain.doFilter(request, response);
    }
}

@Component
public class AuthInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response, Object handler) throws Exception {
        System.out.println("🔐 Interceptor - Checking auth before controller");
        return true; // return false to block the request
    }
}
