*** Explain SOA (Service-Oriented Architecture):

Microservices design patterns:
1. API Gateway Pattern -> A single entry point for all client requests.
	Handles routing, load balancing, authentication, and rate limiting.
	👉 Tools: Spring Cloud Gateway, Kong, AWS API Gateway

2. Service Discovery Pattern -> Automatically register and discover services in dynamic environments
	Tools: Eureka, Consul, Zookeeper
	
3. Circuit Breaker Pattern:
	Prevents a service from repeatedly trying to call a failed service.
	Avoids cascading failures.
	Tools: Resilience4j

4. Retry Pattern:
	Automatically retry a failed request a limited number of times.
	Useful for transient network errors.

5. Database per Service Pattern:
	Each microservice has its own database schema for full decoupling.
	Promotes autonomy and independent scalability.

6. Asynchronous messaging:  Each services communicate with each other through message queues like Kafka or RabbitMQ.

7. Saga Pattern: to manage transactions between multiple microservices
	A saga is a sequence of local transactions. Each service completes its local transaction and 
	then triggers the next step via an event or API call. If any step fails, a compensating transaction 
	is executed to roll back the previous steps.
	
8. CQRS (Command Query Responsibility Segregation)
	Separate read and write models for better scalability and performance.

3. FeignClient (Declarative)
Type: Declarative HTTP client by Netflix (Spring Cloud)
Use Case: Inter-service communication in microservices
It is a declarative REST Client in Spring Boot Web Application. Declarative REST Client means to specify the client specification as an Interface and spring boot will take care of the implementation.
With the help of FeignClient, writing web services is very simple.
It is mostly used to consume REST API endpoints exposed by third parties or microservices.

🔸 Pros:
Clean and declarative syntax

🔸 Cons:
Slightly less control over low-level details (compared to WebClient)
Requires Spring Cloud dependencies

@RequestMapping("/api")
public class UserController {
	@GetMapping("/users/{id}")
    public Mono<ResponseEntity<UserDTO>> getUserById(@PathVariable String id) {
        return myService.fetchUser(id)
            .map(user -> ResponseEntity.ok(user))
            .defaultIfEmpty(ResponseEntity.notFound().build());
    }

What is Client-Side Load Balancing?
Client-side load balancing means that the client (caller) is responsible for choosing which 
server instance to send the request to — not a central load balancer.

How It Works
The client (e.g., a microservice using WebClient or FeignClient) queries a service registry 
		(like Eureka, Consul) for available instances.
It picks one using a load balancing algorithm (like round-robin, random, weighted, etc.).
It sends the request directly to the chosen instance.

| Feature                   | Server-Side (e.g., NGINX) | Client-Side (e.g., Spring Cloud) |
| ------------------------- | ------------------------- | -------------------------------- |
| Who chooses server?       | Load balancer (external)  | Client (internal)                |
| Example                   | NGINX, HAProxy, ALB       | Eureka + Feign/WebClient         |
| Central point of failure  | ✅ Yes                     | ❌ No                             |
| Flexibility               | ❌ Limited                 | ✅ High                           |
| Scaling and routing logic | On load balancer          | On client                        |

How Eureka Server and Client Communicate with Each Other in Java Microservices?
1. Service Registration
		On startup, the client sends a registration request to the Eureka server.
		This includes metadata: service ID, IP, port, health URL, etc.
2. Heartbeat / Lease Renewal:
	Every 30 seconds (by default), the client sends a heartbeat to renew its lease.
	If heartbeat stops, server evicts the instance after 90 seconds (configurable).
3. Service Discovery (Client Fetches Registry):
	Clients periodically pull the list of registered services from the Eureka server.
	This local cache is used for client-side load balancing (e.g., with Ribbon or Spring Cloud 
	LoadBalancer).

Circuit Breaker:
A Circuit Breaker is a design pattern used to prevent cascading failures in a distributed system.
It monitors remote calls and stops trying if the service is failing repeatedly.

Why Use a Circuit Breaker?
	In microservices, if one service becomes slow or unavailable, and others keep calling it:
	It can lead to thread exhaustion, timeouts, and system crashes.
	Circuit breaker helps the system stay resilient by failing fast.

Circuit Breaker States:
1. CLOSED    => Requests flow normally
2. OPEN      => Requests are blocked and fallback is triggered
3. HALF-OPEN => Limited trial requests are allowed to test if the service recovered

@CircuitBreaker(name = "userService", fallbackMethod = "userFallback")
public String getUserDetails(String userId) {
    return restTemplate.getForObject("http://user-service/api/users/" + userId, String.class);
}

public String userFallback(String userId, Throwable t) {
    return "Default User";
}
