Integer values from -128 to 127 are cached by the JVM (IntegerCache).


IntegerCache:
IntegerCache is an internal optimization in Java that caches Integer objects for values in the range 
-128 to 127.


Performance: Avoids creating new objects for commonly used small integers.
Memory Optimization: Saves heap space for frequently repeated values (e.g., loop counters, array indexes).

How it works:
private static class IntegerCache {
    static final int low = -128;
    static final int high = 127;
    static final Integer[] cache;

    static {
        cache = new Integer[(high - low) + 1];
        for (int i = low; i <= high; i++) {
            cache[i + 128] = new Integer(i);
        }
    }
}

public class VolatileTest {
    private static boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        Thread t = new Thread(() -> {
            while (running) {
                // busy-wait
            }
            System.out.println("Thread stopped.");
        });

        t.start();
        Thread.sleep(1000);
        running = false;
        System.out.println("Main thread set running = false");
    }
}

Without volatile:
The running flag is a shared variable, but not volatile.
The JVM may cache the value of running in the thread's local memory (CPU cache).
So Thread t may never see the updated value (false) set by the main thread.
➡️ Infinite loop → "Thread stopped." may never print.

class DeadlockTest {
    private static final Object lock1 = new Object();
    private static final Object lock2 = new Object();

    public static void main1(String[] args) {
        Thread t1 = new Thread(() -> {
            synchronized (lock1) {
                System.out.println("Thread 1: Holding lock1...");

                try { Thread.sleep(100); } catch (InterruptedException e) {}

                System.out.println("Thread 1: Waiting for lock2...");
                synchronized (lock2) {
                    System.out.println("Thread 1: Acquired lock2!");
                }
            }
        });

        Thread t2 = new Thread(() -> {
            synchronized (lock2) {
                System.out.println("Thread 2: Holding lock2...");

                try { Thread.sleep(100); } catch (InterruptedException e) {}

                System.out.println("Thread 2: Waiting for lock1...");
                synchronized (lock1) {
                    System.out.println("Thread 2: Acquired lock1!");
                }
            }
        });

        t1.start();
        t2.start();
    }
}

To avoid deadlock:
Ensure all threads acquire locks in the same order:
synchronized (lock1) {
    synchronized (lock2) {
        // Safe: consistent locking order
    }
}

EnumMap:
| Feature           | `HashMap`        | `EnumMap`                            |
| ----------------- | ---------------- | ------------------------------------ |
| Key type          | Any object       | **Only enum types**                  |
| Performance       | Slower for enums | ✅ Very fast (uses array internally)  |
| Memory efficiency | More overhead    | ✅ Compact (no hashing required)      |
| Null keys allowed | ✅ Yes            | ❌ No (throws `NullPointerException`) |
| Ordering          | Unordered        | ✅ Maintains **natural enum order**   |

🔹 How EnumMap Works Internally

Internally uses an array indexed by the ordinal value of enum constants (0, 1, 2, ...)
So no hashing or collision resolution is needed
That’s why it’s blazing fast and very memory-efficient
