** Cascades:
cascade refers to operations that should be automatically propagated from a parent entity to its 
associated child entities
A cascade tells the JPA provider (e.g., Hibernate) what operations on a parent entity should automatically
 be applied to its related entities.

Types:
| Cascade Type          | Description                                                                      |
| --------------------- | -------------------------------------------------------------------------------- |
| `CascadeType.ALL`     | Applies all cascade types below (save, update, delete, etc.)                     |
| `CascadeType.PERSIST` | When the parent is persisted (saved), the child entity is also persisted         |
| `CascadeType.MERGE`   | When the parent is merged (updated), the child entity is merged                  |
| `CascadeType.REMOVE`  | When the parent is deleted, the child entity is also deleted                     |
| `CascadeType.REFRESH` | When the parent is refreshed from the database, child is also refreshed          |
| `CascadeType.DETACH`  | When the parent is detached from persistence context, the child is also detached |

@Entity
public class User {   
    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL)
    private List<Order> orders;
}
If you save a User, all their orders will be saved.
If you delete a User, all their orders will also be deleted.

** 1. Attached Entity (Managed Entity):
An attached or managed entity is an instance that is currently tracked by the Persistence Context 
(i.e., EntityManager).
This means any changes made to it are automatically detected and synchronized with the database.

Entity is in the managed state.
Inside an active transaction, changes are auto-detected and saved on flush/commit.
You don’t need to explicitly call save() or merge() to persist changes.

User user = entityManager.find(User.class, 1L); // managed
user.setName("New Name"); // automatically tracked
// No need to call save(); it's updated on commit

** 2. Detached Entity:
A detached entity is no longer tracked by the Persistence Context.
Changes made to it are not persisted automatically unless you reattach it using merge().

Outside the persistence context.
Commonly seen after a transaction ends, or after serialization/deserialization (e.g., sending over a network).
You must call merge() to reattach and persist changes.

User user = entityManager.find(User.class, 1L); // managed
entityManager.detach(user); // now detached
user.setName("Detached Name"); // not tracked
entityManager.merge(user); // reattaches and updates DB

** EntityManager.find() and EntityManager.getReference():
EntityManager.find():
	Immediately fetches the entity from the database.
	Returns a fully initialized object.
	If the entity is not found, returns null.
	
EntityManager.getReference()
	Returns a lazy proxy to the entity.
	Does not hit the database immediately.
	The actual database access happens only when a method is called on the proxy.
	Throws EntityNotFoundException if the entity does not exist and is accessed.

*** NamedQuery:

Reusability: named queries can be defined once and used multiple times throughout the application.

Maintenance: named queries can be easily modified or updated in a central location, rather than scattered throughout the codebase.

** Entity States:
| State                    | Description                                                                                                                                   |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------- |
| **New (Transient)**      | The entity has just been created using `new`. It is **not associated** with any persistence context.                                          |
| **Managed (Persistent)** | The entity is attached to the current **`EntityManager`** and is being tracked. Any changes will be automatically **synchronized to the DB**. |
| **Detached**             | The entity was once managed but is now **no longer attached** to a persistence context. Changes **won’t be saved** unless reattached.         |
| **Removed**              | The entity is marked for **deletion**. Will be deleted from DB when the transaction is committed.                                             |

** Optimistic Locking	vs Pessimistic Locking
1. Optimistic Locking:
	“Trust first, check later”
	hink: “I assume no one else will change the data.”
	Multiple users can read the same data at the same time.
	When someone tries to save, JPA checks if the data has changed (using a @Version field).
	If it has changed → throw an error (OptimisticLockException).

	example:
	Two users open a form to edit a product.
	Both make changes.
	User A saves → success.
	User B saves later → fails because data was changed after B read it.

2.Pessimistic Locking:
	“Lock first, then work”
	Think: “I’ll lock this data so no one else can touch it.”
	When one user reads the data, it’s locked in the database.
	Others must wait to read or update it.
	Prevents conflicts before they happen.

	👮 Example:
	User A reads a product for editing.
	Data is locked. Only row is locked..
	User B tries to read/update → must wait or gets an error.

| Feature            | Optimistic Locking        | Pessimistic Locking        |
| ------------------ | ------------------------- | -------------------------- |
| Locking style      | No lock initially         | Locks DB row immediately   |
| Conflict detection | On save/commit            | On read                    |
| Performance        | Faster, scalable          | Slower, more DB load       |
| Best for           | Low-conflict environments | High-conflict environments |
| Error thrown       | `OptimisticLockException` | `PessimisticLockException` |


** Implement Optimistic locking:
@Entity
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    @Version
    private int version;  // used for optimistic locking.. increment and saves in db every time it modify object
	
}

** Implement Pessimistic Locking
public interface ProductRepository extends JpaRepository<Product, Long> {

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdForUpdate(@Param("id") Long id);
}
